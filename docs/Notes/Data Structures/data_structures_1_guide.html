Array vs Linked List: Finding Middle Element

Given a data structure with n elements, find the middle element. Compare the approaches for arrays vs linked lists.

Approach 1: Array Implementation

1
Direct Index Access

Arrays provide O(1) access to any element by index. Finding the middle is trivial:

def find_middle_array(arr): # Get length and calculate middle index n = len(arr) middle_index = n // 2 # Direct access return arr[middle_index] # Example arr = [1, 2, 3, 4, 5] result = find_middle_array(arr) # Returns 3
2
Complexity Analysis

Time: O(1) Space: O(1)

Random access makes this operation constant time - we directly jump to the middle index.

Key Advantage of Arrays

Arrays store elements contiguously in memory, allowing O(1) access to any element through index calculation: base_address + (index √ó element_size)

Approach 2: Linked List Implementation

1
Two-Pointer Technique

Use slow and fast pointers to find the middle without counting:

class Node: def __init__(self, data): self.data = data self.next = None def find_middle_linked_list(head): if not head: return None slow = fast = head # Fast moves 2x speed of slow while fast and fast.next: slow = slow.next fast = fast.next.next # When fast reaches end, slow is at middle return slow.data
2
Why This Works

When the fast pointer reaches the end, it has traveled 2n steps. The slow pointer has traveled n steps, placing it exactly at the middle.

$$ ext{When } fast = ext{end}, ext{ then } slow = rac{n}{2}$$
3
Complexity Analysis

Time: O(n) Space: O(1)

We must traverse half the list, but use only two pointers (constant space).

The two-pointer technique is a common pattern in linked list problems. Master it for interviews!

Approach 3: Comparison & Trade-offs

Performance Comparison

Operation Array Linked List
Find Middle O(1) O(n)
Insert at Beginning O(n) O(1)
Delete at Middle O(n) O(n)
Memory Contiguous Scattered
Cache performance matters! Arrays benefit from spatial locality, making them faster in practice even when Big-O suggests otherwise.

Reversing a Linked List

Reverse a singly linked list. Return the new head of the reversed list.

Approach 1: Iterative (Three Pointers)

1
Setup Three Pointers

Track previous, current, and next nodes:

def reverse_list(head): prev = None current = head while current: # Save next node next_node = current.next # Reverse the link current.next = prev # Move pointers forward prev = current current = next_node # prev is now the new head return prev
2
Visualization

Original: 1 ‚Üí 2 ‚Üí 3 ‚Üí 4 ‚Üí None

After reversal: None ‚Üê 1 ‚Üê 2 ‚Üê 3 ‚Üê 4

Result: 4 ‚Üí 3 ‚Üí 2 ‚Üí 1 ‚Üí None

3
Complexity

Time: O(n) Space: O(1)

Visit each node once, use only three pointers.

Draw diagrams when solving linked list problems - visualizing pointer movements is crucial!

Approach 2: Recursive

1
Recursive Solution

Use the call stack to reverse:

def reverse_list_recursive(head): # Base cases if not head or not head.next: return head # Recursive call new_head = reverse_list_recursive(head.next) # Reverse the link head.next.next = head head.next = None return new_head
2
How It Works

The recursion goes to the end of the list, then unwinds, reversing links on the way back:

  1. Recurse to the last node (new head)
  2. On return, reverse current node's link
  3. Set current.next = None to avoid cycles
  4. Return new head through all calls
3
Complexity

Time: O(n) Space: O(n)

Space is O(n) due to recursion call stack.

Recursive solutions are elegant but use O(n) stack space. For very large lists, iterative is safer.

Approach 3: Stack-Based

1
Using a Stack

Push all nodes onto a stack, then rebuild:

def reverse_list_stack(head): if not head: return None # Push all nodes to stack stack = [] current = head while current: stack.append(current) current = current.next # Pop and rebuild new_head = stack.pop() current = new_head while stack: current.next = stack.pop() current = current.next current.next = None # Important! return new_head
2
Complexity

Time: O(n) Space: O(n)

Two passes through the list, O(n) space for the stack.

When to Use Each Method

Iterative: Best for production - O(1) space, fastest

Recursive: Clean and elegant for interviews, educational

Stack: Demonstrates stack LIFO property, rarely optimal

Detecting Cycles in a Linked List

Determine if a linked list has a cycle. If yes, find the starting node of the cycle.

Approach 1: Floyd's Cycle Detection (Tortoise & Hare)

1
Two-Pointer Technique

Use slow and fast pointers:

def has_cycle(head): if not head or not head.next: return False slow = head fast = head while fast and fast.next: slow = slow.next # Move 1 step fast = fast.next.next # Move 2 steps if slow == fast: return True # Cycle detected! return False # No cycle
2
Why This Works

If there's a cycle, the fast pointer will eventually catch up to the slow pointer inside the cycle. Think of it like runners on a circular track - the faster runner will lap the slower one.

If cycle exists: slow and fast will meet in at most 2 √ó cycle_length iterations
3
Complexity

Time: O(n) Space: O(1)

Optimal solution - linear time, constant space!

Floyd's Cycle Detection

Also called "tortoise and hare algorithm". The key insight: if there's a cycle, a pointer moving at 2x speed will eventually catch a pointer moving at 1x speed.

Approach 2: Hash Set

1
Track Visited Nodes
def has_cycle_hashset(head): visited = set() current = head while current: if current in visited: return True # Cycle found! visited.add(current) current = current.next return False
2
Complexity

Time: O(n) Space: O(n)

Simple but uses extra memory for the set.

Hash set approach is intuitive and easier to code, but Floyd's algorithm is preferred in interviews for O(1) space.

Approach 3: Finding Cycle Start Node

1
Extended Floyd's Algorithm

After detecting cycle, find where it begins:

def detect_cycle_start(head): if not head or not head.next: return None # Phase 1: Detect cycle slow = fast = head while fast and fast.next: slow = slow.next fast = fast.next.next if slow == fast: break else: return None # No cycle # Phase 2: Find cycle start slow = head while slow != fast: slow = slow.next fast = fast.next return slow # Cycle start node
2
Mathematical Proof

Let's prove why this works:

  • Let L = distance from head to cycle start
  • Let C = cycle length
  • When they meet, slow traveled L + a, fast traveled L + a + nC
  • Since fast = 2 √ó slow: L + a + nC = 2(L + a)
  • Simplifying: L = nC - a
  • Therefore, L steps from head = L steps from meeting point = cycle start!
3
Complexity

Time: O(n) Space: O(1)

Beautiful Mathematical Insight

The distance from head to cycle start equals the distance from meeting point to cycle start. This elegant property makes the algorithm work!

Dynamic Array Implementation

Implement a dynamic array that grows automatically when capacity is reached. Include append(), insert(), and delete() operations.

Approach 1: Dynamic Array Class

1
Implementation
class DynamicArray: def __init__(self): self.capacity = 2 self.size = 0 self.array = [None] * self.capacity def _resize(self): # Double the capacity self.capacity *= 2 new_array = [None] * self.capacity # Copy elements for i in range(self.size): new_array[i] = self.array[i] self.array = new_array def append(self, value): # Resize if needed if self.size == self.capacity: self._resize() self.array[self.size] = value self.size += 1 def insert(self, index, value): if index < 0 or index > self.size: raise IndexError("Index out of bounds") if self.size == self.capacity: self._resize() # Shift elements right for i in range(self.size, index, -1): self.array[i] = self.array[i - 1] self.array[index] = value self.size += 1 def delete(self, index): if index < 0 or index >= self.size: raise IndexError("Index out of bounds") # Shift elements left for i in range(index, self.size - 1): self.array[i] = self.array[i + 1] self.size -= 1 self.array[self.size] = None # Clear reference
2
Complexity

Append: O(1) amortized

Insert: O(n)

Delete: O(n)

Approach 2: Amortized Analysis

1
Understanding Amortization

Although resize is O(n), it happens infrequently. Let's analyze the cost:

  • Starting capacity: 2
  • Resize at insertions: 2, 4, 8, 16, 32, ...
  • Cost to resize to size n: n (copy all elements)
  • Total cost for n operations: n + n/2 + n/4 + ... + 2 + 1
2
Geometric Series
$$ ext{Total cost} = n \left(1 + rac{1}{2} + rac{1}{4} + \cdots ight) < 2n$$ $$ ext{Amortized cost per operation} = rac{2n}{n} = O(1)$$

Why Doubling Strategy Works

Doubling capacity ensures that expensive resize operations are rare enough to maintain O(1) amortized time. Growing by a constant factor (not addition) is key!

Summary: Arrays provide O(1) random access but expensive insertion/deletion. Linked lists offer O(1) insertion/deletion but O(n) access. Choose based on your use case!
